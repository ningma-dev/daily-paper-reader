## 摘要
视觉-语言-动作（VLA）策略通常以Python/PyTorch堆栈的形式发布，这些堆栈假设使用工作站级GPU，这与机器人实际运行的硬件不匹配。我们提出了vla.cpp，一个基于llama.cpp构建的可移植C++推理运行时。据我们所知，它是第一个原生支持流匹配和扩散VLA推理模式的ggml类引擎，在该模式中，缓存的视觉-语言前缀由跨注意力动作专家消费，并在多个求解器步骤中集成。一个单一的运行时通过一个请求/响应协议服务于跨越五个骨干网络和四个动作头家族的七种架构，每个模型被打包为一个自包含的bundle。在LIBERO-Object上，该引擎与最先进的检查点在200个episode内仅相差一个episode，并在1.3 GiB内存中以100%的成功率运行BitVLA。同一个bundle在三个硬件层级上无需更改即可运行，从消费级GPU到8 GB的嵌入式模块。跨硬件的屋顶线分析表明，batch-1的VLA推理受计算限制，因此利用率而非带宽是部署的关键；由此分析推导出的IMMA梯形GEMM将BitVLA每步延迟降低了4.5倍。然后，我们在ALOHA机械臂上进行了机器人上的压力测试，该测试分离了学习型VLA必须针对移动目标在其训练的硬件上重新规划的延迟约束。代码、演示视频和可重复基准测试框架可在https://fai-modelopt-tech.github.io/vla-cpp.github.io/获取。

## Abstract
Vision-Language-Action (VLA) policies are typically shipped as Python/PyTorch stacks that assume a workstation-class GPU, a mismatch for the hardware on which robots actually run. We present vla.cpp, a portable C++ inference runtime built on llama.cpp. To our knowledge, it is the first ggml-class engine to natively serve the flow-matching and diffusion VLA inference pattern, in which a cached vision-language prefix is consumed by a cross-attending action expert integrated over several solver steps. A single runtime serves seven architectures spanning five backbone and four action-head families behind one request/response protocol, with each model packaged as a self-contained bundle. On LIBERO-Object, the engine matches a state-of-the-art checkpoint to within one episode out of 200, and runs BitVLA at 100% success in 1.3 GiB of memory. The same bundle runs unchanged across three hardware tiers, from a consumer GPU down to an 8 GB embedded module. A cross-hardware roofline analysis shows that batch-1 VLA inference is compute-bound, so utilization rather than bandwidth is the deployment lever; an IMMA ladder GEMM derived from this analysis cuts BitVLA per-step latency by 4.5x. We then frame an on-robot stress test on an ALOHA arm that isolates the latency constraint under which a learned VLA must replan against a moving target on the hardware it was trained for. Code, demo videos, and the reproducible benchmark scaffold are available at https://fai-modelopt-tech.github.io/vla-cpp.github.io/.

- **问题**：现有的视觉-语言-动作（VLA）策略通常以 Python/PyTorch 堆栈形式发布，依赖工作站级 GPU（如 RTX 3090），但机器人实际部署的硬件往往是内存受限的嵌入式模块（如 Jetson Orin，共享 8 GB 统一内存）。这导致模型难以移植、运行效率低。
- **背景**：大型语言模型的便携推理引擎（如 llama.cpp）已解决 LLM 的部署问题，但它们不支持 VLA 特有的推理模式：① 需要将视觉-语言前缀缓存并跨多个去噪步骤重复使用；② 需要流匹配或扩散动作头。现有方案（如 TensorRT-LLM、MLC-LLM、torch.compile）要么不支持 Jetson Orin，要么不支持动态跨注意力缓存生命周期，要么仅针对单架构优化。
- **核心目标**：构建一个统一的、可移植的 C++ 推理运行时，能够服务多种 VLA 架构，在异构边缘硬件上实现与参考 PyTorch 检查点相同的行为，并分析部署瓶颈。

## 2. 论文提出的方法论：核心思想、关键技术细节

### 核心思想
- 基于 llama.cpp/ggml 构建 C++ 运行时，原生支持 VLA 推理模式：一次预填充编码前缀（compute-bound），多次去噪步骤中重复使用缓存的跨注意力键值（memory-bound）。
- 单个二进制和一种 GGUF 自包含包格式即可服务七种架构，无需重编译。

### 关键技术细节
- **架构设计**：C++ 服务器（加载模型、执行推理）与 Python 客户端（控制循环、处理状态）通过 Protobuf + ZeroMQ 通信，实现语言无关、低延迟。
- **VLA 推理流程**（Algorithm 1）：
  1. 双向注意力编码视觉-语言前缀（图像、指令、本体状态）。
  2. 缓存跨注意力 KV。
  3. 从噪声初始化动作块。
  4. 迭代 T 个求解器步骤：动作头交叉注意力至缓存前缀，更新动作向量。
  5. 反归一化输出动作块。
- **模型支持**：覆盖 5 个视觉-语言骨干（SmolVLM2、PaliGemma、BitNet–SigLIP、InternVL3、Eagle/Cosmos）和 4 个动作头家族（流匹配专家、交叉注意力流匹配、扩散 Transformer、三元组回归）。
- **内核优化**：基于 IMMA（整数矩阵乘法累加）张量核的自定义 W2A8（2-bit 权重，8-bit 激活）GEMM，比 dp4a 内核降低 BitVLA 延迟 4.5×。
- **精度门控**：将位置嵌入索引等离散操作用带进度的精度感知实现，并设置编译时数值一致性检查，防止无声错误。

## 6. 论文的主要结论与发现

1. **行为保真度**：vla.cpp 在 LIBERO-Object 上与参考检查点的成功率差异≤1 个 episode（BitVLA 完美 200/200），所有架构均复现参考行为。
2. **推理加速**：相对 PyTorch eager，SmolVLA 加速 7.95×（28.16 ms vs 223.96 ms），且内存占用相似。
3. **跨硬件约束**：在 8 GB Orin Nano 上，**内存容量是主要瓶颈**，仅 5/7 架构能加载；延迟并非约束。
4. **计算特性**：单请求 VLA 推理中，**预填充阶段是计算受限**（高操作强度），动作专家阶段是**内存受限**。因此，提升计算单元利用率（如 IMMA 内核）比优化带宽更有效。
5. **精度敏感性**：低精度（FP16）下离散位置嵌入索引的舍入差异可无声地致残任务（动作位移达 1.97 σ，任务成功从 90% 降至 20%），需通过精度感知实现和数值一致性检查来防御。
6. **真实机器人验证**：更低的推理延迟（470 ms vs 620 ms）使观察更新更及时，ALOHA 成功率从 40% 提升至 87.5%，任务时间缩短。

## 7. 优点

- **高可移植性**：单一二进制 + 统一 GGUF 包格式，跨从消费 GPU 到 8 GB 嵌入式模块的硬件无需重编译。
- **广泛架构支持**：首个在同一引擎中支持 5 种骨干 + 4 种动作头家族的 ggml 类运行时。
- **首个端到端三元组 VLA**：在引擎中完整运行 BitVLA（1.58-bit 权重、视觉塔和动作头），包含自定义 IMMA GEMM。

- **实用部署工具**：包含精度门控、客户端-服务器解耦、可重复基准框架。

## 8. 不足与局限

- **缺乏动态权重分载机制**：大模型（如 GR00T-N1.6/N1.7）在 8GB Nano 上无法直接加载，需要将模拟器卸载到另一台机器。
